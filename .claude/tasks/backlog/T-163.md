## Description

Deploy two independent TWIN Nodes locally using Docker Compose for DSP protocol support (Phase 3). Not needed for Phase 2 (in-process connectors provide identity and notarization directly).

### Architecture

```
docker-compose.twin.yml

  db-alpha (MySQL 8.4.7, port 3306)
  twin-bootstrap-alpha (one-shot, creates DID Alpha)
  twin-node-alpha (port 3000)

  db-beta (MySQL 8.4.7, port 3307)
  twin-bootstrap-beta (one-shot, creates DID Beta)
  twin-node-beta (port 3001)
```

### When this becomes necessary

- DSP Federated Catalogue requires persistent, discoverable HTTPS endpoints
- ODRL enforcement (PEP/PDP) is built into the TWIN Node rights management service
- W3C Activity Streams data plane requires server-to-server HTTP POST

### Acceptance criteria

- Two TWIN Nodes running on localhost:3000 and localhost:3001
- Each has its own DID identity (different `did:iota:testnet:0x...`)
- Session and data isolation confirmed
- `docker compose -f docker-compose.twin.yml up` starts everything
- `ADAPTER_MODE=twin-node` routes to deployed TWIN Nodes
- Total resource footprint: < 2GB RAM, < 5GB disk
