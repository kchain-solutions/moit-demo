## Description

Validate that `@iota/notarization` and `@iota/iota-interaction-ts` WASM bindings load and execute correctly on the development platform (macOS ARM) and target deployment platform (Linux x64).

### Test script

Create `server/twin/test-wasm.ts`:

```typescript
import { NotarizationClient, start } from '@iota/notarization/node';
import { Ed25519KeypairSigner } from '@iota/iota-interaction-ts/node/test_utils';

async function testWasm() {
  start(); // Initialize WASM runtime
  console.log('WASM runtime initialized');

  // Verify key types are available
  console.log('NotarizationClient:', typeof NotarizationClient);
  console.log('Ed25519KeypairSigner:', typeof Ed25519KeypairSigner);

  // Verify getChainIdentifier monkey-patch works
  // (twin-etd-poc uses this to work around a WASM binding issue)
}

testWasm().then(() => console.log('WASM test passed'));
```

### Known issues from twin-etd-poc

- `getChainIdentifier()` monkey-patch in `iotaNotarizationHelper.ts` (line 28-35)
- `start()` must be called before any WASM operations
- Some WASM types require specific Node.js flags (check if `--experimental-wasm-modules` needed)

### Acceptance criteria

- WASM bindings load on macOS ARM (dev)
- WASM bindings load on Linux x64 (deploy)
- `NotarizationClient.create()` can be instantiated
- Document any platform-specific workarounds
