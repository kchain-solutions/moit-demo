## Description

Define the `ITwinAdapter` interface and implement `InProcessAdapter` wrapping `server/twin/connectors.ts` and `server/twin/notarization.ts`.

### ITwinAdapter interface

```typescript
interface ITwinAdapter {
  init(mnemonic: string): Promise<boolean>;
  isReady(): boolean;

  // Identity
  createDid(partyName: string): Promise<{ did: string; walletAddress: string; explorerUrl: string }>;
  resolveDid(did: string): Promise<unknown>;
  getCachedDids(): Record<string, string>;

  // Notarization
  createNotarization(digestSRI: string, documentId: string, documentTypeCode: string, issuerDid: string): Promise<string | undefined>;
  verifyNotarization(notarizationId: string, expectedDigestSRI: string): Promise<boolean>;

  // Proofs
  createProof(did: string, document: Record<string, unknown>): Promise<unknown>;
  verifyProof(document: Record<string, unknown>, proof: unknown): Promise<boolean>;
}
```

### Implementations

1. **SimulatedAdapter** (`server/adapter/simulated.js`): Returns fake DIDs, random hashes, simulated verification. No network calls. Current demo behavior.

2. **InProcessAdapter** (`server/adapter/in-process.js`): Wraps `server/twin/connectors.ts` and `server/twin/notarization.ts`. Each call executes in-process via WASM bindings to IOTA testnet. Auto-funds wallets via faucet.

3. **TwinNodeAdapter** (`server/adapter/twin-node.js`): Planned for Phase 3. REST client to deployed TWIN Node.

### Acceptance criteria

- `ITwinAdapter` interface defined
- `SimulatedAdapter` produces identical behavior to current demo
- `InProcessAdapter` creates real DIDs and notarizations when `IOTA_MNEMONIC` is set
- Route handlers call `adapter.createDid()`, `adapter.createNotarization()` etc. without knowing the backend
