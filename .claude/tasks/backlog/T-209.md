## Description

`ADAPTER_MODE=simulated|in-process|twin-node`. Controls which `ITwinAdapter` implementation is used at startup.

### Modes

- **`simulated`** (default): Current demo behavior. `SimulatedAdapter` returns fake DIDs, random hashes. No network calls. No `IOTA_MNEMONIC` required. Suitable for offline development, CI/CD, and presentations without internet.

- **`in-process`**: Real IOTA operations via npm packages. `InProcessAdapter` wraps `server/twin/connectors.ts`. Requires `IOTA_MNEMONIC` env var (BIP-39 seed phrase). Auto-funds wallet from testnet faucet.

- **`twin-node`** (Phase 3): REST client to deployed TWIN Node. `TwinNodeAdapter` calls `TWIN_NODE_URL` endpoints. Requires Docker deployment (T-163). Needed for DSP protocol support.

### Implementation

```javascript
// server/adapter/index.js
import { SimulatedAdapter } from './simulated.js';
import { InProcessAdapter } from './in-process.js';

const ADAPTER_MODE = process.env.ADAPTER_MODE || 'simulated';

export function createAdapter() {
  switch (ADAPTER_MODE) {
    case 'in-process': return new InProcessAdapter();
    case 'twin-node': throw new Error('twin-node mode available in Phase 3');
    default: return new SimulatedAdapter();
  }
}
```

### Acceptance criteria

- `ADAPTER_MODE=simulated` produces identical behavior to current demo
- `ADAPTER_MODE=in-process` initializes TWIN connectors and creates real DIDs
- Missing `IOTA_MNEMONIC` with `in-process` mode logs a clear error
- Mode is logged at server startup
